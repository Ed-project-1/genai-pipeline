EXECUTIVE SUMMARY
================================================================================

This meeting was convened to finalize the Q3 AI infrastructure roadmap,
approve budget allocations for the generative AI platform, review security
and compliance postures, and align cross-functional teams on delivery
milestones. Key decisions include the adoption of AWS Bedrock as the primary
LLM provider, approval of a $2.4M infrastructure budget, and the green-light
for Phase 1 of the internal AI assistant ("Project Meridian").

================================================================================
AGENDA ITEM 1: OPENING REMARKS & STRATEGIC CONTEXT
================================================================================

Sarah Mitchell opened the meeting at 9:04 AM. She noted the company is at a
critical inflection point: three major enterprise competitors have already
deployed internal GenAI tools, and customer surveys (Q2 2024) indicate 67%
of enterprise clients now expect AI-augmented services as a baseline offering.

She emphasized that this roadmap must balance:
  (a) Speed-to-market for competitive parity
  (b) Security and data governance — particularly given upcoming EU AI Act
      compliance requirements effective March 2025
  (c) Cost discipline — the board has set a hard ceiling of $3M for H2 2024
      AI infrastructure spend

Sarah introduced the three strategic pillars for the AI buildout:
  1. INTERNAL PRODUCTIVITY — GenAI tools for employees (summarization,
     search, code assistance, meeting intelligence)
  2. CUSTOMER-FACING PRODUCTS — AI-powered features embedded in the
     core product suite (smart recommendations, document Q&A, auto-reports)
  3. DATA PLATFORM MODERNIZATION — RAG pipelines, vector databases, and
     a unified data layer to support AI workloads at scale

AGENDA ITEM 2: TECHNICAL ARCHITECTURE REVIEW
================================================================================

Dr. James Okonkwo presented the proposed architecture diagram (shared screen,
slides available in Confluence: /AI-Roadmap/Q3-2024/Architecture).

KEY ARCHITECTURAL DECISIONS:

  2.1 LLM Provider Selection
  ---------------------------
  After a 6-week evaluation of OpenAI (Azure), Google Vertex AI, AWS Bedrock,
  and Cohere, the team recommends AWS Bedrock as the primary LLM provider.

  Evaluation criteria and scores (1–10):
    Criteria                | Azure/OpenAI | Vertex AI | AWS Bedrock | Cohere
    ------------------------|--------------|-----------|-------------|-------
    Data residency controls |     6        |    7      |     9       |   7
    Enterprise SLA          |     8        |    7      |     9       |   6
    Model variety           |     9        |    8      |     8       |   5
    Cost at scale           |     5        |    6      |     8       |   8
    IAM / VPC integration   |     4        |    5      |     10      |   4
    Compliance certs        |     7        |    7      |     9       |   6
    Langchain compatibility |     9        |    8      |     8       |   7

  AWS Bedrock advantages:
  - Native IAM integration — no external API keys stored in secrets
  - VPC endpoints — all traffic stays within the AWS network
  - SOC 2 Type II, ISO 27001, HIPAA eligible — matches our compliance needs
  - Supports Claude 3, Llama 3.1, Mistral, Titan, and Cohere natively
  - Bedrock Knowledge Bases integrates natively with OpenSearch Serverless

  Primary model selected: Anthropic Claude 3.5 Sonnet (bedrock alias)
  Fallback model: Meta Llama 3.1 70B Instruct (cost-optimized tasks)

  2.2 Vector Database Selection
  ------------------------------
  Decision: Amazon OpenSearch Serverless (vector engine mode)

  Alternatives considered:
    - Pinecone: Excellent performance but external SaaS — data residency concern
    - Weaviate (self-hosted on EKS): High operational overhead
    - pgvector on Aurora: Good for small scale; may bottleneck at >10M vectors
    - OpenSearch Serverless: Native AWS, no infra to manage, auto-scales,
      supports ANN (approximate nearest neighbor) with NMSLIB and FAISS engines

  Marcus Webb noted that LangChain has a first-class OpenSearch integration
  via `langchain_community.vectorstores.OpenSearchVectorSearch`, which
  simplifies the retrieval chain implementation significantly.

  Vector dimensions: 1536 (Titan Embeddings v2) or 768 (Cohere Embed)
  Selected embedding model: Amazon Titan Embeddings V2 (bedrock)
  Chunking strategy: Recursive character text splitter, chunk_size=1000,
  chunk_overlap=200

  2.3 Application Layer
  ----------------------
  Framework: LangChain (Python) — v0.2.x
  API Layer: FastAPI on AWS Lambda (containerized via Lambda Container Images)
  Orchestration: LangChain LCEL (LangChain Expression Language) pipelines
  Agents: LangChain AgentExecutor with custom tools:
    - DocumentSearchTool (vector retrieval)
    - SummaryTool (map-reduce summarization)
    - MetadataFilterTool (OpenSearch metadata filters)
    - CalendarTool (stub — Phase 2)

  2.4 Infrastructure as Code
  ---------------------------
  Tool: AWS CloudFormation (YAML templates)
  Deployment pipeline: GitHub Actions CI/CD
  Environments: dev → staging → prod (separate AWS accounts via Organizations)
  Secrets management: AWS Secrets Manager + Parameter Store
  Logging: CloudWatch Logs + X-Ray distributed tracing
  Storage: S3 (raw documents), S3 (processed/chunked), DynamoDB (session state)

  Angela Torres raised the question of whether to use CDK instead of raw
  CloudFormation. After discussion, the team agreed raw CloudFormation YAML
  is preferred because:
    - The infrastructure team has more CF expertise
    - CDK adds a build step that complicates the GitHub Actions pipeline
    - CF templates are more auditable for compliance reviews
    - CDK can always be adopted in Phase 3 if needed

AGENDA ITEM 3: PROJECT MERIDIAN — PHASE 1 SCOPE
================================================================================

Priya Sharma walked through the Phase 1 product scope.

Project Meridian is an internal AI assistant that allows employees to:
  1. Upload meeting memos, PDFs, Word documents, and Confluence exports
  2. Ask natural-language questions against the document corpus
  3. Generate summaries, action item extracts, and status reports
  4. Receive source-cited answers with document provenance

Phase 1 deliverables (target: November 30, 2024):
  - Document ingestion pipeline (S3 → Lambda → chunking → embeddings → OpenSearch)
  - RAG Q&A endpoint (FastAPI + LangChain + Bedrock)
  - Simple chat UI (React, internal only)
  - Admin panel for document management
  - GitHub Actions CI/CD pipeline
  - Full CloudFormation stack (VPC, Lambda, OpenSearch, S3, IAM)

Out of scope for Phase 1:
  - Multi-tenant isolation (Phase 2)
  - Real-time document streaming
  - Fine-tuning or model customization
  - Integration with Slack / Teams
  - Customer-facing deployment

Nina Patel shared UX research findings: internal testers (n=24) rated the
prototype Q&A interface 4.2/5 for usefulness but noted confusion around
source citations. She recommended surfacing document name, date, and page
number alongside each answer. The team agreed to include this in Phase 1.

AGENDA ITEM 4: SECURITY & COMPLIANCE REVIEW
================================================================================

Daniel Kovacs presented the security architecture.

  4.1 Data Classification
  ------------------------
  Documents ingested into Meridian are classified as:
    - INTERNAL (default) — accessible to all employees
    - CONFIDENTIAL — restricted to designated teams (HR, Legal, Finance)
    - SECRET — executive-only; scoped access via IAM conditions

  Metadata tagging at ingest time will encode classification level.
  LangChain retrieval filters will enforce classification gates at query time.

  4.2 Network Security
  ---------------------
  - All Lambda functions deployed inside a private VPC
  - OpenSearch domain accessible only via VPC endpoint
  - Bedrock API calls routed via VPC interface endpoint (no internet egress)
  - S3 access via VPC gateway endpoint
  - No public internet exposure for any backend service
  - API Gateway (public-facing) sits in front of Lambda, with Cognito auth

  4.3 Identity & Access
  ----------------------
  - Users authenticate via Cognito User Pools (SSO via existing Okta IdP)
  - Lambda execution roles follow least-privilege principle
  - Bedrock model access gated via resource-based IAM policies
  - All access logs shipped to CloudWatch; SIEM integration in Phase 2

  4.4 EU AI Act Compliance Notes (from Legal, written input)
  -----------------------------------------------------------
  The EU AI Act classifies internal productivity AI tools as "limited risk."
  Requirements applicable to Phase 1:
    - Transparency notice to employees that AI is processing their queries
    - Human oversight mechanism (feedback buttons, escalation path)
    - Data retention policy: query logs purged after 90 days
    - No automated decision-making with legal or HR implications in Phase 1

  Daniel confirmed the architecture meets these requirements. A formal
  compliance attestation document will be prepared by Legal before launch.
